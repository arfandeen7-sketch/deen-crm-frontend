## 1. Overview

### Tech stack
- **Runtime:** Node.js `>=24`, TypeScript, CommonJS
- **Framework:** Express `5.1`
- **ORM:** Prisma `6.1` → PostgreSQL (Supabase-hosted)
- **Auth:** Supabase Auth (`@supabase/supabase-js` admin client) — JWT bearer tokens
- **File handling:** `multer` (memory storage), `exceljs` (import/export), `pdfkit` (payslips)
- **Email:** `nodemailer` (SMTP)
- **Scheduling:** `node-cron`
- **Validation:** `zod`
- **Security:** `helmet`, `cors`, `express-rate-limit`
- **External APIs:** `axios` (Facebook Graph), `google-ads-api`, `ua-parser-js`

### High-level request flow
```
route file → (app-level authenticate) → (route-level authorize RBAC) → asyncHandler(controller)
   → controller validates body with Zod → calls Prisma (and/or service layer) → res.json(...)
   → errors bubble to global errorHandler middleware
```
Source: `src/app.ts`, `src/middleware/*`, `src/utils/asyncHandler.ts`.

### Base URL / versioning
- All API routes are mounted under **`/api`**. There is **no version segment** (no `/v1`).
- Server port: `PORT` env or `3001` (`src/index.ts`).
- Health endpoints (no `/api` prefix, no auth):
  - `GET /` → `{ "message": "Welcome to DEEN CRM Backend" }`
  - `GET /health` → `{ "status": "OK", "timestamp": "<ISO>" }`

### Global success envelope
There are **two** conventions in use (be aware — this is not perfectly uniform):

1. **Single-resource / action responses** — wrapped:
   ```json
   { "data": <object|array|primitive-wrapper> }
   ```
   Used by nearly all controllers (e.g. `res.json({ data: lead })`).

2. **Paginated list responses** — `data` plus sibling meta fields (NOT a nested `meta` object). Built by `paginatedResponse()` in `src/utils/pagination.ts`:
   ```json
   {
     "data": [ ...items ],
     "total": 123,
     "page": 1,
     "pageSize": 25,
     "totalPages": 5
   }
   ```

   ⚠️ Note: `dashboard`, `users.list`, `leave.balance`, etc. return aggregate objects inside `{ data: {...} }`, NOT the paginated shape. Each endpoint below states which it uses.

3. **Binary responses** (exports, PDFs): raw file body with `Content-Disposition` + `Content-Type` headers, no JSON envelope.

### Global error format (`src/middleware/error.middleware.ts`)
- **Zod validation failure → `400`:**
  ```json
  {
    "error": "Validation failed",
    "details": [ { "path": "email", "message": "Invalid email" } ]
  }
  ```
- **Multer upload error → `400`:** `{ "error": "File too large. Maximum upload size is 5 MB" }` (or `"Upload error: <msg>"`)
- **`HttpError` (thrown explicitly) → its status:** `{ "error": "<message>" }`
- **Any other error:** uses `err.status`/`err.statusCode` if present, else `500`. For `>=500` the message is masked to `"Internal Server Error"`.

So the **standard error shape is `{ "error": string }`**, with an optional `details[]` array only for Zod failures.

### Auth mechanism summary
- Client sends `Authorization: Bearer <supabase_access_token>`.
- `authenticate` middleware calls `supabase.auth.getUser(token)` (network call per request), then loads the matching Prisma `User` by `id` and attaches it to `req.user`.
- Failure modes: missing/invalid header → `401 "Missing or invalid Authorization header"`; bad token → `401 "Unauthorized"`; no profile → `401 "User profile not found"`; inactive account → `403 "Account deactivated"`.

## 2. Authentication & Authorization

### Token
- **Type:** Supabase-issued JWT (access token from `signInWithPassword`).
- **Payload:** managed by Supabase (not custom-minted here). The backend does **not** read roles from the token — it derives role/identity from the Prisma `User` row matched by the Supabase user `id` (UUID).
- The Prisma `User.id` **is** the Supabase `auth.users` UUID (mirrored on user creation).

### Roles (`UserRole` enum)
- `master` — full access (super admin)
- `hr_manager` — HR/attendance/payroll/employees
- `sales_manager` — leads/brokers management
- `sales_executive` — restricted to **own** assigned leads / own HR records

### How protected routes signal required roles
- `authorize(...roles)` factory (`src/middleware/rbac.middleware.ts`) returns `403 "Insufficient permissions"` if `req.user.role` is not in the allowed list.
- Applied either **router-wide** (`router.use(authorize(...))`) or **per-route**.
- Many controllers additionally enforce **row-level scoping** in code (e.g. a `sales_executive` is force-scoped to `assignedTo === req.user.id` regardless of query params).

### Auth endpoints (`/api/auth`)
| Method | Path | Auth | Purpose |
|---|---|---|---|
| POST | `/api/auth/login` | public (rate-limited 10/15min) | Email+password login |
| POST | `/api/auth/logout` | authenticated | Sign out current token |
| PUT | `/api/auth/change-password` | authenticated | Change own password |

**POST `/api/auth/login`**
- Body (Zod): `{ email: string(email), password: string(min 1) }`
- Success `200`: `{ data: { token: string, user: { id, fullName, email, role } } }`
- Errors: `401 "Invalid email or password"`, `401 "User profile not found"`, `403 "Account deactivated"`, `400` Zod, `429` rate limit (`{ error: "Too many attempts, please try again later" }`).

**POST `/api/auth/logout`**
- No body. Success `200`: `{ data: { success: true } }`. (Best-effort Supabase sign-out.)

**PUT `/api/auth/change-password`**
- Body: `{ newPassword: string(min 8) }`
- Success `200`: `{ data: { success: true } }`; Errors: `400` (Supabase rejection or Zod).

## 4. API Endpoints

> Unless stated, all endpoints below require a valid JWT (`authenticate`). Validation errors → `400` Zod shape. Not-found → `404 { error }`. Forbidden role → `403 { error: "Insufficient permissions" }`.

### 4.1 Dashboard (`/api/dashboard`)
Scoping: `sales_executive` is scoped to own assigned leads; others see all.

**GET `/api/dashboard/summary`** — lead counts by source.
- Response `200`: `{ data: { sourceCounts: [{ source: string, count: number }], totalLeads: number } }` (sourceCounts sorted desc by count).

**GET `/api/dashboard/status-analytics`** — status update activity over last 24h.
- Response `200`: `{ data: { analytics: [{ status: string, leadCount: number, updateCount: number }], windowHours: 24 } }`.

### 4.2 Leads (`/api/leads`)
Row-level scoping: `sales_executive` always restricted to `assignedTo === self`; cannot see/edit others' leads (`403`).

**GET `/api/leads`** — paginated list.
- Query: `page`, `pageSize` (10/25/50/100, default 25); filters: `assignedTo` (ignored for SE), `source`, `status`, `serviceType`, `projectName`, `city`, `locality`, `dateFrom`, `dateTo` (filter `leadDate`), `search` (matches `leadName` OR `mobileNumber`), `category` ∈ `fresh|imported|assigned|unassigned|untouched`.
- Response: **paginated envelope**; each item includes `assignedUser: { id, fullName }` and full `broker` object.

**GET `/api/leads/:id`**
- Response `200`: `{ data: Lead & { assignedUser: { id, fullName }, broker, statusHistory: LeadStatusHistory[] (desc) } }`.
- `404` if not found; `403` if SE and not owner.

**POST `/api/leads`** — create (any authenticated user).
- Body (Zod `createSchema`): required `leadName`, `mobileNumber`, `source`, `serviceType`; optional `alternateMobile`, `projectName`, `email` (email or `""`), `city`, `locality`, `comments`, `leadStatus`, `leadPriority`, `followUpDate` (coerced date), `assignedTo` (uuid), `brokerId` (uuid).
- Side effects: validates `source`/`leadStatus`/`leadPriority` against `dynamic_fields` (`400 Invalid <category>: "<value>"` on miss); sets `isTouched=false`, `ingestionSource=manual`, `createdBy=self`.
- Response `201`: `{ data: Lead }`.

**PUT `/api/leads/:id`** — update.
- Body: all create fields optional (`createSchema.partial()`).
- Side effects: if `leadStatus` changes → inserts a `LeadStatusHistory` row (in a transaction). Sets `isTouched=true` if any of `leadStatus`/`followUpDate`/`assignedTo`/`comments` provided ("meaningful" update; never resets to false).
- `404`/`403` as above. Response `200`: `{ data: Lead }`.

**DELETE `/api/leads/:id`** — `authorize('master')`.
- `404` if not found. Response `200`: `{ data: { success: true } }`.

**POST `/api/leads/bulk-assign`** — `authorize('master','sales_manager')`.
- Body: `{ leadIds: uuid[] (min 1), assignedTo: uuid }`.
- Sets `assignedTo` + `isTouched=true` on all. Response: `{ data: { updated: number } }`.

**POST `/api/leads/bulk-status`** — `authorize('master','sales_manager')`.
- Body: `{ leadIds: uuid[] (min 1), status: string }`.
- Updates status + inserts history rows for those whose status actually changed. Response: `{ data: { updated: number } }` (count = number of leads matched).

**POST `/api/leads/import`** — `authorize('master','sales_manager')`, multipart field `file` (`.xlsx`/`.csv`, ≤5MB).
- Expected columns: `lead_name`, `mobile_number`, `alternate_mobile`, `source`, `service_type`, `project_name`, `email`, `city`, `locality`, `comments`, `lead_status`, `lead_priority`.
- Required per row: `lead_name`, `mobile_number`, `source`, `service_type`. Duplicate `mobile_number` (any existing lead) → skipped.
- Response: `{ data: { imported: number, skipped: number, errors: [{ row: number, reason: string }] } }`.
- Errors: `400` no file / invalid type / too large.

**GET `/api/leads/export`** — XLSX of filtered leads (uses same query filters as list). Binary response.

**GET `/api/leads/template`** — blank XLSX import template. Binary response.

**GET `/api/leads/report`** — `authorize('master')`. Aggregated report.
- Query (Zod): `groupBy` ∈ `user|source` (default `user`), `dateFrom`, `dateTo` (filter `createdAt`), `userId` (uuid), `source`.
- Response `200`: `{ data: { groupBy: string, rows: ReportRow[] } }` where `ReportRow = { groupKey, groupLabel, totalAssigned, touched, untouched, followedUp, missedFollowUps, statusBreakdown: Record<string,number>, lastActivityAt: Date|null }` (sorted desc by totalAssigned).

**GET `/api/leads/report/export`** — `authorize('master')`. Same data flattened to XLSX (status columns prefixed `status_`). Binary.

### 4.3 Follow-ups (`/api/followup`)
Scoping: SE scoped to own. Compares `followUpDate` to today (UTC midnight).

| Method | Path | Bucket logic |
|---|---|---|
| GET | `/api/followup/today` | `followUpDate` in [today, tomorrow) |
| GET | `/api/followup/missed` | `followUpDate` < today |
| GET | `/api/followup/upcoming` | `followUpDate` >= tomorrow |

- Query: `page`, `pageSize`. Response: **paginated envelope**, each item includes `assignedUser: { id, fullName }`. Order: `upcoming` asc, others desc.

### 4.4 Brokers (`/api/brokers`)

**GET `/api/brokers`** — list. Query: `page`, `pageSize`, `search` (brokerName OR mobileNumber), `status`. Paginated envelope.

**GET `/api/brokers/:id`** — `{ data: Broker }`; `404` if missing.

**GET `/api/brokers/:id/leads`** — paginated leads for broker (SE scoped to own). Paginated envelope.

**POST `/api/brokers`** — `authorize('master','sales_manager')`.
- Body: required `brokerName`, `mobileNumber`; optional `companyName`, `status` ∈ `active|inactive|suspended`. Sets `postedBy=self`. Response `201`: `{ data: Broker }`.

**PUT `/api/brokers/:id`** — `authorize('master','sales_manager')`. Body partial. `404` if missing. `{ data: Broker }`.

**DELETE `/api/brokers/:id`** — `authorize('master')`. `{ data: { success: true } }`.

**POST `/api/brokers/import`** — `authorize('master')`, multipart `file`.
- Columns: `broker_name`, `mobile_number`, `company_name`, `status`. Required: `broker_name`, `mobile_number`. Dup mobile skipped. Response: `{ data: { imported, skipped, errors:[{row,reason}] } }`.

**GET `/api/brokers/export`** — `authorize('master','sales_manager')`. XLSX. Binary.

### 4.5 Users (`/api/users`) — entire router `authorize('master')`

**GET `/api/users`** — `{ data: { users: User[], roleCounts: Record<role, number> } }` (NOT paginated).

**GET `/api/users/:id`** — `{ data: User }`; `404` if missing.

**POST `/api/users`** — creates Supabase auth user then Prisma profile.
- Body: `{ fullName, email, password(min 8), phone?, role? (default sales_executive) }`.
- Side effects: creates Supabase auth user (`email_confirm: true`); on Prisma failure, **rolls back** the orphaned auth user. `400` if Supabase rejects. Response `201`: `{ data: User }`.

**PUT `/api/users/:id`** — Body: `{ fullName?, phone?, role? }`. `{ data: User }`.

**PATCH `/api/users/:id/toggle-active`** — flips `isActive`. `{ data: User }`.

### 4.6 Employees (`/api/employees`) — router `authorize('master','hr_manager')`
Operates on the **same `users` table** but exposes HR attributes.

**GET `/api/employees`** — list. Query: `page`, `pageSize`, `department`, `designation`, `status` (employmentStatus), `role`, `search` (fullName/email/employeeId). Paginated envelope (items are full `User` rows).

**GET `/api/employees/export`** — XLSX of all users. Binary.

**GET `/api/employees/:id`** — `{ data: User }`; `404`.

**POST `/api/employees`** — creates auth user + full HR profile.
- Body: `fullName`, `email`, `password(min 8)`, `phone?`, `role?`, `employeeId?`, `department?`, `designation?`, `joiningDate?`, `basicSalary?` (>=0), `allowances?` (>=0), `leaveBalance?` (Record<string, number>), `bankName?`, `bankAccountNumber?`, `bankIban?`, `employmentStatus?`.
- Validates `department`/`designation`/`bankName` against `dynamic_fields`. Rolls back auth user on failure. Response `201`: `{ data: User }`.

**PUT `/api/employees/:id`** — Body: create fields minus `email`/`password`, all optional. Validates dynamic fields. `{ data: User }`.

**PATCH `/api/employees/:id/status`** — Body: `{ status: EmploymentStatus }`. `{ data: User }`.

### 4.7 Attendance (`/api/attendance`)
Privileged = `master`|`hr_manager` (see all + manual edits). Others scoped to own.

**POST `/api/attendance/check-in`** — own record; multipart `photo` **required** (or `photo` base64 data-URI in body).
- Photo: JPEG/PNG/WebP, ≤5MB. `409` if already checked in today.
- Side effects: uploads photo to Supabase Storage; computes `status` from check-in time (UAE): `<09:45 present`, `09:45–10:30 late`, `>10:30 half_day`. Upserts the day's record.
- Response `201`: `{ data: Attendance & { checkInPhotoUrl: string|null } }`.

**POST `/api/attendance/check-out`** — own record; `photo` required.
- `409` if no check-in / already checked out. Computes `totalWorkingHours`. Response `200`: `{ data: Attendance & { checkOutPhotoUrl } }`.

### 4.8 Leave Requests (`/api/leave`)
Privileged = `master`|`hr_manager`. Others scoped to own.

**GET `/api/leave`** — list. Query: `page`, `pageSize`, `userId` (privileged), `status`, `leaveType`. Paginated; items include `user: { id, fullName }`.

**GET `/api/leave/balance`** — own balance, or `?userId=` for privileged. Response: `{ data: { userId, fullName, leaveBalance: object } }`. `404` if user missing.

**GET `/api/leave/:id`** — `{ data: LeaveRequest & { user:{id,fullName} } }`. `404`; `403` if not owner/privileged.

**POST `/api/leave`** — apply (any authenticated user, for self).
- Body: `{ leaveType ∈ annual|sick|emergency|unpaid, dateFrom (date), dateTo (date), reason? }`.
- Validation: `dateTo >= dateFrom` (`400`); range must contain ≥1 non-Sunday working day (`400`); rejects if overlapping pending/approved request exists (`409`).
- `totalDays` computed = non-Sunday days in range. Response `201`: `{ data: LeaveRequest }` (status `pending`).

**PUT `/api/leave/:id/review`** — `authorize('master','hr_manager')`.
- Body: `{ status ∈ approved|rejected, reviewNote? }`. `409` if request not `pending`.
- Side effects on **approve** (transaction): backfills `Attendance` rows (`status=leave`, `leaveType`) for every working day in range; for paid types (`annual|sick|emergency`) **decrements** `user.leaveBalance[leaveType]` by `totalDays` (floored at 0).
- Response `200`: `{ data: LeaveRequest }`.

**DELETE `/api/leave/:id`** — cancel **own pending** request only. `403` if not owner; `409` if not pending. `{ data: { success: true } }`.

## 5. Webhooks & External Integrations

All ingested leads flow through `ingestLead()` (`src/services/ingestion/ingestion.utils.ts`), which:
- Dedups by `externalLeadId` (unique), then by same `mobileNumber + source` within **7 days**.
- Sets defaults: `serviceType="Buy"`, `leadStatus="Fresh"`, `createdBy` = first `master` user (system actor; throws if none exists).
- Stores the full raw payload in `Lead.rawPayload`.

### Facebook Lead Ads / Instagram
- **GET verify:** query `hub.mode=subscribe`, `hub.verify_token` (must match `FACEBOOK_WEBHOOK_VERIFY_TOKEN`), `hub.challenge` → echoes challenge with `200`, else `403`.
- **POST:** verifies `X-Hub-Signature-256` HMAC-SHA256 against raw body using `FACEBOOK_APP_SECRET` (`401` on mismatch). Responds `200 { received: true }` immediately, then processes async.
- Incoming shape: `{ object: "page", entry: [{ changes: [{ field: "leadgen", value: { leadgen_id, form_id, ad_id? } }] }] }`.
- Backend then **calls Graph API** to fetch `field_data` for the lead and the form name.
- Mapping → Lead: `leadName` ← `full_name`/`name`; `mobileNumber` ← `phone_number`/`phone`/`mobile`; `email` ← `email`; `city` ← `city`; `comments` ← `comments`/`message`; `fbFormName` ← form name; `source` = `Instagram` if `ad_id` contains "instagram" else `Facebook`; `ingestionSource` = `instagram`/`facebook`; `externalLeadId` = `leadgen_id`.

### Google Lead Form Extensions
- **POST `/google/webhook`:** authenticated by `body.google_key === GOOGLE_WEBHOOK_SECRET` (`401` on mismatch). Responds `200 { received: true }`, processes async.
- Incoming shape: `{ lead_id?, user_column_data: [{ column_name, string_value }] }`.
- Mapping: `leadName` ← `full_name`; `mobileNumber` ← `phone_number`; `email` ← `email`; `source="Google"`; `externalLeadId` ← `lead_id` (or `google_<ts>`).
- Also polled via cron from the Google Ads API (`lead_form_submission_data`), mapping field types `FULL_NAME`/`FIRST_NAME`+`LAST_NAME`, `PHONE_NUMBER`, `EMAIL`, `CITY`.

### Property Finder
- **POST `/property-finder/webhook`:** verifies `x-pf-signature` HMAC-SHA256 against raw body using `PROPERTY_FINDER_WEBHOOK_SECRET` (`401` on mismatch). Responds `200 { received: true }`, processes async.
- Incoming shape (⚠️ VERIFY — exact PF schema unconfirmed in code; mapping is defensive): reads `payload.lead ?? payload`, then `name|full_name`, `phone|mobile`, `email`, `emirate|city`, `message|note`, `id|lead_id`.
- `source="Property Finder"`, `ingestionSource=property_finder`, `externalLeadId` ← `id`/`lead_id` (or `pf_<ts>`).

> ⚠️ Raw body capture for FB + Property Finder requires `express.raw` before `express.json` (configured in `app.ts`). Google uses parsed JSON (no signature, key-in-body).

## 6. Background Jobs (`src/jobs/leadSync.job.ts`)

Started on server boot (`startLeadSyncJobs()` in `index.ts`). **Must run on a single instance.**

| Schedule (cron) | Job | What it does | Models touched |
|---|---|---|---|
| `*/15 * * * *` (every 15 min) | Facebook poll | Fallback poll for FB leads since cursor; advances `SyncCursor[facebook]` on success | `Lead`, `SyncCursor` |
| `*/10 * * * *` (every 10 min) | Google poll | Polls Google Ads `lead_form_submission_data` since cursor; advances `SyncCursor[google]` | `Lead`, `SyncCursor` |
| `55 19 * * *` (19:55 UTC = 23:55 UAE) | Attendance reconciliation | For every active user (`isActive` + `employmentStatus=active`) with **no** attendance record today, creates one with `status=absent` | `Attendance`, reads `User` |

Cursors persist via the `sync_cursors` table; default lookback is 1 hour if no cursor.

## 7. Business Logic Notes Relevant to Frontend

- **Time zone:** Attendance "today" and status use **UAE time (UTC+4, no DST)**. `@db.Date` fields are stored as UTC-midnight of the UAE day.
- **Attendance status from check-in time (UAE):** `<09:45` → `present`; `09:45–10:30` → `late`; `>10:30` → `half_day`. No record by night → `absent` (reconciliation job).
- **Working hours:** computed at check-out = `(checkOut - checkIn)` hours, 2-decimal.
- **Attendance photos:** required on check-in/check-out. Stored as private paths; responses include short-lived (1h) signed URLs `checkInPhotoUrl`/`checkOutPhotoUrl`. If `ATTENDANCE_PHOTO_BUCKET` unset, paths/URLs are `null`.
- **Leave working days:** `totalDays` = count of **non-Sunday** days in `[dateFrom, dateTo]`. Sundays are the weekend. (No multi-day weekend / holiday calendar in code.)
- **Leave approval state machine:** `pending` → `approved` | `rejected` (only `pending` is reviewable). User may `cancel` (delete) **only** own `pending` requests. Approval backfills attendance (`status=leave`) and decrements `leaveBalance` for paid types (`annual`, `sick`, `emergency`); `unpaid` does not deduct balance.
- **Payroll computation (`computeForUser`):**
  - `presentDays` = count of `present` + `late`.
  - `halfDays` = count of `half_day` (counted as 0.5 day earned).
  - `approvedLeaveDays` = `leave` records with a paid `leaveType`.
  - `unpaidLeaveDays` = `absent` records + `leave` with `unpaid` type.
  - `workingDaysInMonth` = non-Sunday days in month.
  - `perDayRate = basicSalary / workingDaysInMonth`.
  - `earnedDays = presentDays + approvedLeaveDays + halfDays*0.5`.
  - `deductions = max(0, basicSalary - earnedDays*perDayRate)`.
  - `netSalary = earnedDays*perDayRate + allowances + overtimeAmount`.
- **`isTouched` / "untouched leads":** a lead is *untouched* until a meaningful update (status change, follow-up date, assignment, or comment) flips `isTouched=true`. Bulk-assign also sets it true. Never reset to false. The `category=untouched` filter returns `isTouched=false`.
- **Lead assignment:** `assignedTo` is nullable. `sales_executive` only ever sees/edits their own assigned leads (server enforces, ignoring `assignedTo` query for that role). `category=unassigned` → `assignedTo=null`.
- **Status history:** every status change (single update or bulk) inserts a `LeadStatusHistory` row; powers the 24h dashboard analytics.
- **Seeded dynamic fields** (`prisma/seed.ts`):
  - `lead_status`: Fresh, Existing Client, Cold, Interested, No Answer Msg Dropped, Not Interested, Not Receiving Calls.
  - `lead_priority`: Hot, Warm, Cold.
  - `source`: Property Finder, Instagram, Facebook, Google, Manual, Import.
  - (Other categories like `department`, `designation`, `bank_name`, `project_name`, etc. have **no seed data** — must be created via the dynamic-fields API before use, or create/update of leads/employees referencing them will `400`.)
- **Pagination:** `pageSize` must be one of `10|25|50|100` (anything else → default `25`); `page` defaults to 1. List responses are the flat envelope `{ data, total, page, pageSize, totalPages }` (no nested `meta`).
- **Decimal/Date serialization:** Prisma `Decimal` → JSON **string**; dates → ISO strings. Frontend should parse numbers from salary/hours strings.
- **Two logout flows:** `/api/auth/logout` (Supabase token sign-out) and `/api/login-activity/logout` (session-log close) are independent.

## 8. Known Gaps / Incomplete Areas

- **Candidate/HR module unimplemented & unmounted:** `src/routes/hr.routes.ts` defines `/candidates` CRUD + export + bulk-assign, but (a) it is **not mounted** in `app.ts`, and (b) every handler in `hr.controller.ts` returns `501 { error: "Not implemented" }`. No `Candidate` model exists. **Frontend should treat all `/candidates` endpoints as non-functional.**
- **No Shift model / shift endpoints:** Despite references to office hours and a commented-out `Shift` seed in `prisma/seed.ts`, there is **no `Shift` model**, no shift routes/controllers. Office hours are hardcoded constants in `attendance.service.ts`.
- **`PayslipStatus.generated` never set:** code only writes `draft` and `sent`. The intermediate `generated` enum value is unused.
- **Property Finder payload unconfirmed:** field mapping is defensive/guessed (see Section 5) — exact incoming schema is **not** verified in code (flagged in source comments). **⚠️ VERIFY.**
- **Holidays/weekend calendar:** only Sunday is treated as a non-working day; `weekend`/`holiday` attendance statuses exist in the enum but are never set automatically by any code path. **⚠️ VERIFY** how they get populated (likely manual override only).
- **Dedup may suppress genuine re-inquiries:** the 7-day mobile+source dedup window in `ingestLead` is flagged in code as a business decision to confirm.
- **No `meta` envelope:** the API does NOT use a `{ data, meta }` envelope; pagination metadata is at the top level. Plan/spec naming may differ from implementation.
- **Auth per-request network cost:** `authenticate` calls Supabase `getUser()` on every request (noted in code as a future optimization to local JWKS verification).
- **`leaveBalance` shape is free-form JSON:** no schema enforces its keys; code reads/writes `leaveBalance[leaveType]`. Frontend should treat it as `Record<string, number>` keyed by leave type. **⚠️ VERIFY** seeding/initialization of balances (no default set on user create unless provided).
